# Cleave HHJP ビルドガイド総レビュー

レビュー日: 2026-05-21

対象: [build-guide/README.md](build-guide/README.md), [build-guide/01-ASSEMBLY.md](build-guide/01-ASSEMBLY.md), [build-guide/02-FIRMWARE.md](build-guide/02-FIRMWARE.md), [build-guide/03-CASE.md](build-guide/03-CASE.md), [build-guide/04-USAGE.md](build-guide/04-USAGE.md), [build-guide/05-CUSTOMIZE.md](build-guide/05-CUSTOMIZE.md)

実施方針: サブエージェントで「情報構造」と「表現方法」を分けて点検したうえで、本文・見出し・画像参照・リンクを再確認しました。修正は行わず、レビュー結果のみをまとめています。

## 総評

全体として、購入者が「部品確認、はんだ付け、ファームウェア書き込み、ケース組み立て、利用、カスタマイズ」まで進むための主要情報はかなり揃っています。写真、完了チェック、トラブルシューティング、ZMK Studio と GitHub Actions の使い分けがあり、ビルドガイドとしての土台は強いです。

現時点で残る主な課題は、空の導通テスト章、ショットキーダイオード向き情報の未完成、完成後確認で失敗した場合の導線、ペアリング関連情報の整理です。

## 段階1: 情報構造設計

### 高優先度

#### 1. [01-ASSEMBLY.md](build-guide/01-ASSEMBLY.md) に空の章が残っている

[01-ASSEMBLY.md](build-guide/01-ASSEMBLY.md#L244) に「テスターを使った導通テスト」という見出しがありますが、本文がありません。目次にも掲載されているため、読者は重要な確認手順が抜けていると感じます。

推奨:

- 導通テストを必須にするなら、どこを測るか、正常値は何か、異常時にどの章へ戻るかを書くとよいです。
- 任意確認に留めるなら、目次から外すか「必要に応じた確認」として [01-ASSEMBLY.md](build-guide/01-ASSEMBLY.md#L177) の XIAO 周辺ショート確認と統合するとよいです。

#### 2. ショットキーダイオードの向き情報が未完成

[01-ASSEMBLY.md](build-guide/01-ASSEMBLY.md#L135) のショットキーダイオードは左右で向きが違う重要箇所ですが、[01-ASSEMBLY.md](build-guide/01-ASSEMBLY.md#L147) に写真追加予定のプレースホルダが残っています。ビルド失敗や破損につながる可能性があるため優先度が高いです。

推奨:

- 左右別の拡大写真、またはシルクと部品の線を強調した図解を追加するのがよいです。
- 現在の [build-guide/images/assembly/04/1.jpg](build-guide/images/assembly/04/1.jpg) は完成状態の写真としては使えますが、向き確認の主役としては余白と対象外領域が多く、左右差の説明には不足します。

#### 3. 完成後確認とトラブルシューティングの導線が弱い

[04-USAGE.md](build-guide/04-USAGE.md#L13) のチェックリストは実用的ですが、失敗した場合にどこへ戻るかが書かれていません。たとえば Bluetooth に出ない場合は [02-FIRMWARE.md](build-guide/02-FIRMWARE.md#L93)、キーが反応しない場合は [02-FIRMWARE.md](build-guide/02-FIRMWARE.md#L107) へ戻るべきですが、読者が自力で探す必要があります。

推奨:

- チェックリストの後に「うまくいかない場合」の短い対応表を置くとよいです。
- 例: 「Bluetooth に表示されない -> 02 の Bluetooth に表示されない」「一部キーが反応しない -> 02 の一部のキーが反応しない」

### 中優先度

#### 4. ペアリング関連情報が複数章に散っている

ペアリング情報は [02-FIRMWARE.md](build-guide/02-FIRMWARE.md#L61)、[04-USAGE.md](build-guide/04-USAGE.md#L65)、[05-CUSTOMIZE.md](build-guide/05-CUSTOMIZE.md#L175) にあります。章ごとの役割は理解できますが、「左右間ペアリング」「ホスト機器との Bluetooth プロファイル」「ZMK Studio 設定リセット」が混同されやすいです。

推奨:

- [04-USAGE.md](build-guide/04-USAGE.md) に「リセットの種類」早見表を置くとよいです。
- 例: 「ホストとの接続をやり直す: BT_CLR」「左右・永続化設定を初期化する: settings_reset」「ZMK Studio の保存内容を消す: Restore Stock Settings」
- [02-FIRMWARE.md](build-guide/02-FIRMWARE.md#L61) は初回セットアップに絞り、日常運用は [04-USAGE.md](build-guide/04-USAGE.md) へ寄せると構造が締まります。

#### 5. [04-USAGE.md](build-guide/04-USAGE.md) のキーマップ変更章は実質リンク集になっている

[04-USAGE.md](build-guide/04-USAGE.md#L95) の「キーマップ変更」は [05-CUSTOMIZE.md](build-guide/05-CUSTOMIZE.md) への誘導だけです。目次に独立章として載せるほどの内容は少なく、読者にとっては少し重複感があります。

推奨:

- 「基本操作」の一部として短く扱うか、章を残すなら「まずは ZMK Studio、詳しくは 05」のように判断情報を足すとよいです。

#### 6. 3Dプリント材料選択は詳しいが、初心者向けの推奨が埋もれている

[03-CASE.md](build-guide/03-CASE.md#L89) の材料表は丁寧ですが、初めて発注する読者には候補が多く、どれを選ぶべきか判断しにくいです。TIP で PA12-HP Nylon 推奨はありますが、表の後ろにあるため意思決定の入口としては弱めです。

推奨:

- 表の前に「迷ったらこれ」という推奨構成を置くとよいです。
- 例: 「ベースケース: PA12-HP Nylon」「トッププレート: 好みのレジンまたはナイロン」「透明トップカバー: レジン」など。
- 発注サービス名を出すなら、標準ユースケースなのか一例なのかも明記するとよいです。

#### 7. カスタマイズ章は良いが、読まなくてよい読者への出口がない

[05-CUSTOMIZE.md](build-guide/05-CUSTOMIZE.md#L16) は ZMK Studio とフォークの比較が分かりやすいです。一方で、初回ビルド読者に対して「デフォルトのまま使うならここは不要」という前置きがありません。

推奨:

- 冒頭に「初期キーマップで使う場合、この章は読み飛ばせます」を追加すると、読者の認知負荷が下がります。

#### 8. 開発者向け情報への導線はあるが、位置付けを分けたい

[05-CUSTOMIZE.md](build-guide/05-CUSTOMIZE.md#L223) には [docs/firmware_design.md](docs/firmware_design.md) へのリンクがあります。設計書は 1000 行を超える詳細資料で、一般ビルド読者には重いです。

推奨:

- 「参考リンク」内で「開発者向け」または「内部設計を知りたい方向け」と明記するとよいです。
- ビルドガイド本文には必要最小限だけ置き、詳細は設計書に逃がす現方針は維持してよいです。

### 低優先度

#### 9. root README と build-guide README の役割は整理できる

[README.md](README.md#L3) と [build-guide/README.md](build-guide/README.md#L1) はどちらもドキュメント案内ですが、現状は大きな問題ではありません。将来的には root はプロジェクト入口、build-guide は購入者向け入口として役割を明示するとさらに分かりやすくなります。

## 段階2: 表現方法

### 高優先度

#### 1. 画像の未完成・情報不足が信頼感を落としている

最重要は [01-ASSEMBLY.md](build-guide/01-ASSEMBLY.md#L147) の写真追加予定です。ビルドガイド全体は写真が豊富なぶん、未完成の表が目立ちます。

推奨:

- プレースホルダを公開版に残さないほうがよいです。
- 写真がまだない場合は、一時的に図解やシルクの説明で代替し、「写真追加予定」という制作都合は削除するほうが読者向けです。

#### 2. 画像の表示幅指定が不統一

画像表示は `300px`, `400px`, `500px`, `600px`, `800px`, `100%`, 指定なしが混在しています。実寸も 781px から 7008px まで幅があります。読みやすさの観点では、意図のあるサイズ差と単なるばらつきが区別しにくいです。

根拠:

- [01-ASSEMBLY.md](build-guide/01-ASSEMBLY.md#L151) は 300px、[01-ASSEMBLY.md](build-guide/01-ASSEMBLY.md#L78) は 600px です。
- [03-CASE.md](build-guide/03-CASE.md#L29) は 800px、[03-CASE.md](build-guide/03-CASE.md#L179) は 100% です。
- 表示幅のばらつきは、現本文内の画像指定だけでも確認できます。

推奨:

- 「全体写真は 600px」「部品比較は 400px」「2列比較は 100%」「構造図は 800px」など、簡単な基準を決めるとよいです。
- 横長写真はトリミングまたは拡大対象の強調を検討すると、スマホでも読みやすくなります。

#### 3. 画像の代替テキストがない

Markdown 画像ではなく HTML の `<img>` タグが多用されており、alt テキストがありません。視覚的には問題なくても、アクセシビリティ、検索性、画像が表示されない環境での理解に弱くなります。

推奨:

- すべてに詳細な説明を入れる必要はありませんが、手順理解に必須の画像には `alt` を付けるとよいです。
- 例: 「左基板のショットキーダイオード向き」「XIAO をピンヘッダで位置合わせした状態」「トッププレート固定ネジ位置」など。

#### 4. 写真の主役が分かりにくい箇所がある

いくつかの写真は品質は高いものの、対象物の周囲が広く写っており、読者がどこを見るべきか迷います。特に [build-guide/images/assembly/04/1.jpg](build-guide/images/assembly/04/1.jpg) はショットキーダイオード確認に使うには主役が小さく、余白が大きいです。

推奨:

- 重要箇所はトリミング、丸囲み、矢印、左右ラベルを追加するとよいです。
- すでに [build-guide/images/case/07/position.png](build-guide/images/case/07/position.png) のような赤丸付き図は有効なので、この表現をダイオードやバッテリー極性にも展開できます。

### 中優先度

#### 5. 文体はおおむね丁寧だが、文末の役割が少し揺れている

全体は「します」「してください」が中心で読みやすいです。ただし、説明文、指示文、推奨文が近い文体で並ぶため、必須作業と任意作業の差が少し曖昧です。

例:

- [01-ASSEMBLY.md](build-guide/01-ASSEMBLY.md#L19) のテスターは「必須ではありませんが、あると便利」です。
- [01-ASSEMBLY.md](build-guide/01-ASSEMBLY.md#L177) のショート確認は「推奨します」です。
- [03-CASE.md](build-guide/03-CASE.md#L256) の滑り止めは「お好みに応じて」です。

推奨:

- 必須: 「してください」
- 推奨: 「推奨します」または「できれば確認してください」
- 任意: 「必要に応じて」
- 危険回避: 「必ず」「接続しないでください」

このように文末ルールを軽く決めると、チェックリストとの対応も取りやすくなります。

#### 6. 用語表記に小さな揺れがある

大きな支障はありませんが、読み手の迷いを減らすなら統一したい表記があります。

候補:

- 「基板」と「基盤」: [01-ASSEMBLY.md](build-guide/01-ASSEMBLY.md#L55) では見出しが「基板」ですが、本文に「基盤」もあります。
- 「ハンダ付け」と「はんだ付け」: 見出しは「はんだ付け」、本文に「ハンダ付け」が混在します。
- 「オモテ面」「ウラ面」と「表面」「背面」: [01-ASSEMBLY.md](build-guide/01-ASSEMBLY.md#L57), [01-ASSEMBLY.md](build-guide/01-ASSEMBLY.md#L179) などで混在します。
- 「左側キーボード」「左側基板」「左側」: [05-CUSTOMIZE.md](build-guide/05-CUSTOMIZE.md#L44) など。

推奨:

- はんだ作業では「基板」「はんだ付け」「オモテ面/ウラ面」などに統一するとよいです。
- ZMK Studio の対象は「左側基板」または「左側キーボード」のどちらかに寄せるとよいです。

#### 7. コールアウトの重要度ルールを統一したい

NOTE, TIP, IMPORTANT, CAUTION, WARNING が使われています。現状でも読めますが、危険度の基準が明文化されていないため、どの情報が人身・発火リスクで、どの情報が作業ミス防止なのかがやや曖昧です。

推奨:

- WARNING/CAUTION: LiPo、ショート、破損、発火など安全上の注意
- IMPORTANT: 向き、左右差、設定消失など作業成否に関わる注意
- NOTE: 補足
- TIP: おすすめ、コツ

#### 8. 表にしたほうがよい情報がいくつかある

現状でも表はうまく使われていますが、さらに表化すると伝わりやすい箇所があります。

候補:

- [01-ASSEMBLY.md](build-guide/01-ASSEMBLY.md#L247) のバッテリー接続: 「作業前状態」「接続方向」「確認項目」「失敗時」
- [04-USAGE.md](build-guide/04-USAGE.md#L13) の動作確認: 「確認できない症状」と「参照先」
- [04-USAGE.md](build-guide/04-USAGE.md#L65) のリセット: 「BT_CLR」「settings_reset」「Restore Stock Settings」の違い

#### 9. 英語ラベルの図と本文の言語が混在している

[03-CASE.md](build-guide/03-CASE.md#L29) の構成図は英語ラベルです。図そのものは分かりやすいですが、本文が日本語なので初心者には「Top cover」「Base case」などが少し距離を作ります。

推奨:

- 図の直下に日本語対応表を置くか、本文で「トップカバー（Top cover）」のように初出だけ併記するとよいです。
- 図を作り直す必要は必ずしもありません。

### 低優先度

#### 10. 完了チェックの粒度をそろえたい

完了チェックは非常に有効です。一方で、章によって必須確認、任意確認、トラブル予防が混ざっています。

推奨:

- 任意項目には「必要に応じて」「推奨」を入れるとよいです。
- 安全上の必須項目はチェックリストの上部に置くとよいです。

#### 11. 目次と章見出しの粒度を調整したい

[01-ASSEMBLY.md](build-guide/01-ASSEMBLY.md#L244) の空章は別として、[03-CASE.md](build-guide/03-CASE.md#L265) の「13. 完成状態」は作業ステップというより参考写真です。番号付きステップに含めると、作業項目に見えます。

推奨:

- 「参考: 完成状態」または「完成状態の確認」にすると、作業ステップとの違いが明確になります。

#### 12. 画像の近さはおおむね良いが、手順と画像が一対一でない箇所がある

XIAO の位置合わせは [01-ASSEMBLY.md](build-guide/01-ASSEMBLY.md#L168) の表で分かりやすいです。一方で、ケースのいくつかの工程は説明文の後に写真が 1 枚だけあり、確認ポイントが写真内のどこかを探す必要があります。

推奨:

- 重要な作業だけでよいので、画像キャプションに「見るべき点」を 1 行足すとよいです。
- 例: 「赤丸が M2 ネジ位置」「ノブの突起が基板上の電源スイッチに噛み合う向き」など。

## 保持したい強み

- 部品表が具体的で、数量・キット付属・備考が揃っています。
- 完了チェックが多く、読者が進捗確認しやすいです。
- [02-FIRMWARE.md](build-guide/02-FIRMWARE.md#L75) と [05-CUSTOMIZE.md](build-guide/05-CUSTOMIZE.md#L187) のトラブルシューティングは実用的です。
- [05-CUSTOMIZE.md](build-guide/05-CUSTOMIZE.md#L16) の ZMK Studio とフォークの比較は、読者のスキル差に対応できています。
- 画像が豊富で、特に XIAO 実装の段階写真は誤作業防止に役立ちます。
- ケース材料やモデル選択まで踏み込んでいて、単なる組み立て手順以上の情報があります。

## 判断が必要な事項

以下は、残る改善を進める場合に判断が必要です。

1. 3Dプリントサービス利用を標準ユースケースとして扱うか、あくまで例として扱うか。
2. 導通テスト章を復活させるか、任意確認として既存ステップ内へ統合するか。
3. docs のファームウェア設計書を一般ユーザーにも見せるか、開発者向け参考として明確に分けるか。

## 優先度順の対応案

1. [01-ASSEMBLY.md](build-guide/01-ASSEMBLY.md#L244) の空章と [01-ASSEMBLY.md](build-guide/01-ASSEMBLY.md#L147) の写真プレースホルダを解消する。
2. [04-USAGE.md](build-guide/04-USAGE.md#L13) にチェック失敗時の参照先を追加する。
3. ペアリング・リセット種別の説明を整理する。
4. 用語表記と文末ルールを揃える。
5. 画像サイズ、alt、キャプション、強調表示を整える。
